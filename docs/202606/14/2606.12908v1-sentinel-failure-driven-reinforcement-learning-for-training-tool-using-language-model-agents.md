## 摘要
语言模型代理通过多轮工具使用在解决现实任务方面越来越有效。然而，训练可靠的工具使用代理在实践中仍然具有挑战性。虽然强化学习提供了一种从代理自身环境交互中改进代理的在策略范式，但其有效性在很大程度上取决于训练任务分布。当任务在训练前固定时，任务分布可能会与策略不断变化的能力越来越不匹配，导致许多轨迹被浪费在无信息的任务上。我们提出SENTINEL，一种失败驱动的强化学习框架，将求解器的轨迹失败转化为有针对性的训练任务。SENTINEL遵循控制器-提议者-求解器循环：控制器分析失败的轨迹并总结重复出现的错误模式，提议者生成可执行的任务以强调这些弱点，求解器在有针对性的任务上进行训练。在Tau2-Bench Retail上使用Qwen3-4B-Thinking-2507，SENTINEL将Pass^1从66.4提高到74.9，并且在所有Pass^k指标上优于一般合成任务上的强化学习。这些结果表明，模型失败为改进工具使用语言模型代理提供了有效且可扩展的有针对性训练信号来源。

## Abstract
Language model agents are increasingly effective in solving realistic tasks through multi-turn tool use. However, training reliable tool-using agents remains challenging in practice. While reinforcement learning provides an on-policy paradigm for improving agents from their own environment interactions, its effectiveness depends heavily on the training task distribution. When tasks are fixed before training, the task distribution can become increasingly mismatched with the policy's evolving capabilities, causing many rollouts to be spent on uninformative tasks. We propose SENTINEL, a failure-driven reinforcement learning framework that turns the Solver's rollout failures into targeted training tasks. SENTINEL follows a Controller--Proposer--Solver loop: the Controller analyzes failed trajectories and summarizes recurring error patterns, the Proposer generates executable tasks that stress these weaknesses, and the Solver is trained on the targeted tasks. On Tau2-Bench Retail with Qwen3-4B-Thinking-2507, SENTINEL improves Pass\^{}1 from 66.4 to 74.9 and outperforms RL on general synthetic tasks across Pass\^{}k metrics. These results demonstrate that model failures provide an effective and scalable source of targeted training signal for improving tool-using language model agents.

- **问题**：训练工具使用语言模型（LLM）代理时，现有强化学习（RL）方法依赖**固定训练任务集**。随着策略能力演化，任务分布与策略弱点逐渐不匹配，导致大量轨迹浪费在无信息任务上，RL效率低下。
- **核心目标**：提出一种**失败驱动**的RL框架，使训练任务能动态适应当前策略的弱点，通过将代理自身的失败轨迹转化为针对性训练信号，提升工具使用代理的可靠性。
- **背景**：工具使用代理需学会选择正确工具、操作正确记录、遵循多步骤流程和领域政策。监督微调（SFT）是离策略方法，RL虽提供在策略训练，但任务固定的缺陷使其效果受限。

## 2. 论文提出的方法论
### 核心思想：Controller–Proposer–Solver 闭环
- **角色**：
  - **Solver**（求解器）：当前策略模型，通过与环境交互产生轨迹，并基于GRPO（Group Relative Policy Optimization）进行RL优化。
  - **Controller**（控制器）：分析Solver的失败轨迹，提取重复错误模式，并生成指导指令传递给Proposer。
  - **Proposer**（提议者）：根据Controller指令，生成**可执行的、针对性训练任务**，注入任务缓冲区。

### 关键技术细节
1. **失败轨迹分析**：
   - Controller每固定批次触发，对失败轨迹去重，分析错误模式（如错误物品选择、缺失写操作、多记录混淆等）。
   - 输出自然语言指令 `dt`，包含当前错误模式、成功轨迹信息以及历史训练记录。

2. **任务生成**：
   - **轨迹接地（Trajectory-grounded）**：Proposer先在环境中构造一条可执行工具调用轨迹，确保所有中间状态可达。
   - **失败感知重写（Failure-aware rewriting）**：将轨迹重写为用户请求，并刻意加入导致失败的特征（如相似产品变体、需要确认的步骤）。

3. **RL训练与奖励塑造**：
   - 使用GRPO更新策略，奖励函数包括：
     - `Rtask`（任务成功信号）：检查最终环境状态和通信是否满足要求。
     - **行动级检查**：验证中间步骤是否正确（如身份认证）。
     - **惩罚项**：重复工具调用、过长推理、格式错误等，防止奖励黑客。
   - 训练持续迭代，新失败返回Controller，驱动下一轮任务生成。

## 7. 优点
- **动态自适应**：通过失败驱动打破固定任务分布瓶颈，使RL始终针对模型当前弱点，提高样本效率。
- **闭环可控制**：Controller→Proposer→Solver形成可解释的反馈循环，训练方向可监控和调节。
- **奖励塑造精细**：行动级检查和多惩罚项减少奖励黑客，促进可靠工具使用行为。
- **实验验证充分**：不仅对比基线，还测试了SFT后模型场景，展示了方法在多个模型状态下的有效性，避免了仅对弱模型有效的质疑。
