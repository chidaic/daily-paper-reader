## 摘要
生成一个游戏并不等同于制作一个可玩的游戏。尽管代码生成取得了进展，现有方法将游戏生成视为从提示到工件的一次性转换，导致交互层面的故障未被检测到。我们认为，评估和改进游戏生成需要一个玩家，并研究了图形用户界面（GUI）代理在此过程中的两个角色：（1）作为客观评估者，为此我们引入了PlaytestArena，这是一个新的评估环境，将8种类型的200个基于浏览器的游戏生成任务与预期的游戏内行为评分标准配对，由加载每个构建并在浏览器中播放的GUI代理进行裁决；（2）作为主观测试者，为此我们提出了Play2Code，其中游戏代理和GUI代理在共享内存的持续循环中运行，将游戏生成转变为编码与游戏之间的对话。我们的实验表明，即使是前沿模型也难以直接生成可玩的游戏，而Play2Code达到了66.8%的评分标准通过率，分别比单次生成和代理编码基线提高了37.1和14.6个百分点。进一步分析表明，GUI测试者的反馈比人类报告更可追踪，但在某些方面具有类似人类测试者的特性，将游戏测试建立为交互式代码生成的关键测试平台。我们的项目网站位于https://continual-game-generation.vercel.app/。

## Abstract
Generating a game is not the same as making one that can be played. Despite advances in code generation, existing approaches treat game generation as one-shot translation from prompt to artifact, leaving interaction-level failures undetected. We argue that evaluating and improving game generation requires a player, and study two roles for graphical user interface (GUI) agents in this process: (1) as an objective evaluator, for which we introduce PlaytestArena, a new evaluation environment that pairs 200 browser-based game generation tasks across eight genres with rubrics of expected in-play behaviors, adjudicated by a GUI agent that loads each build in a browser and plays it; and (2) as a subjective playtester, for which we propose Play2Code, where a game agent and a GUI agent operate in a sustained loop with shared memory, turning game generation into a dialogue between coding and playing. Our experiments show that even frontier models struggle to generate playable games directly, while Play2Code achieves a 66.8\% rubric pass-rate, improving over single-pass and agentic-coding baselines by 37.1 and 14.6 points respectively. Further analysis shows that GUI playtester feedback is more traceable than a human report, yet idiosyncratic in ways reminiscent of human testers, establishing game playtesting as a critical testbed for interactive code generation. Our project website is available at https://continual-game-generation.vercel.app/.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有游戏生成方法将游戏生成视为从自然语言提示到代码工件的一次性翻译（one-shot），忽略了游戏的可玩性（playability）和交互层的故障检测。代码可以编译运行，但游戏可能在交互层面存在缺陷（如按钮无响应、精灵不渲染、胜利条件不触发），这些缺陷无法通过静态分析或编译检查发现。
- **研究动机**：游戏的质量存在于玩家的实际体验中，而非代码本身。因此，评估和改进游戏生成需要一个“玩家”角色。论文提出利用图形用户界面（GUI）代理作为人类测试者的替代，承担客观评估和主观测试双重角色。
- **整体含义**：将游戏生成从单次翻译转变为持续的“编码-游戏”对话，通过GUI代理的反馈驱动游戏迭代进化，从而提升生成游戏的可玩性和质量。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想
- 引入GUI代理作为游戏生成的“玩家”，承担两个角色：
  1. **客观评估者**：通过PlaytestArena评估生成游戏，根据预定义的评分标准（rubric）自动判定游戏是否满足预期行为。
  2. **主观测试者**：在Play2Code框架中，GUI代理与游戏代理形成闭环，通过共享记忆系统持续反馈，驱动游戏代码的迭代优化。

### 2.2 关键技术细节
- **PlaytestArena**：
  - 包含200个浏览器游戏生成任务，覆盖8种类型（谜题、策略、卡牌、平台游戏、动作、管理、射击、其他）。
  - 每个任务配备一个自然语言生成提示和一个评分标准（rubric），包含5个维度：机制、控制、进度、界面、视觉反馈，共1,548条标准（平均每游戏7.7条）。
  - 评分过程：GUI代理加载游戏、在浏览器中实际游玩，观察渲染界面，通过点击和按键操作，并根据观察到的行为判定每条标准通过与否，最终通过率作为游戏得分。
- **Play2Code框架**：
  - 由**游戏代理（Game Agent）** 和**GUI代理（GUI Agent）** 组成，围绕共享游戏运行时和三层记忆系统协同工作。
  - **游戏代理**负责生成/修补游戏代码（HTML/CSS/JS），输入包括当前代码、GUI反馈和记忆系统。
  - **GUI代理**负责加载游戏并游玩，输出游戏摘要和可操作修复列表（如“敌人保持静止而非巡逻→检查敌人巡逻触发条件”）。
  - **循环流程**：游戏代理生成/修补代码 → GUI代理加载并游玩 → 输出反馈 → 游戏代理读取反馈并更新代码 → 进入下一轮。循环终止条件：GUI代理判定游戏已达到提示意图，或达到最大轮数（5轮）。
  - **记忆系统**：三层结构。
    - **Episode记忆**：任务内记忆，存储回合历史、摘要、修复、尝试。
    - **Skill记忆**：跨任务、代理隔离的记忆，存储每个代理积累的专长（游戏代理：代码模式；GUI代理：交互策略）。
    - **World记忆**：跨任务、共享记忆，存储通用游戏规则、设计原则等。
  - **技术关键**：GUI代理不访问代码或DOM，仅通过渲染界面感知和操作，确保反馈基于实际体验。

## 6. 论文的主要结论与发现

1. **直接生成可玩游戏困难**：即使前沿模型一次性生成的游戏通过率也很低（平均约29.7%）。
2. **Play2Code显著提升生成质量**：在三个骨干上平均rubric通过率达到66.8%，比Direct LLM提升37.1个百分点，比OpenGame提升14.6个百分点。
3. **游戏复杂度影响改进效果**：低复杂度游戏得分快速收敛；中等复杂度游戏稳定提升；高复杂度游戏提升有限，瓶颈在于GUI代理的游玩能力而非循环本身。
4. **GUI代理反馈可追溯且类人**：GUI代理的反馈锚定在可恢复的动作序列上，比人类报告更可追踪；不同骨干代理产生的反馈侧重点不同（如Kimi K2.5偏重视觉美学），反映类似人类的“品味”差异。
5. **GUI代理作为可靠评估器**：GUI裁判与人类注释者在逐条标准上的一致率84.2%（Cohen's κ=0.64），游戏级排名相关系数0.87，接近人类间一致性，证明其可作为自动化评估工具。
